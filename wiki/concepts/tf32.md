# TF32 (TensorFloat32)

NVIDIA's 19-bit floating point format introduced with Ampere architecture. Provides near-FP32 stability with FP16-like speed, requiring zero code changes.

## Format Specification

| Format | Sign | Exponent | Mantissa | Total Bits |
|--------|------|----------|----------|------------|
| FP32 | 1 | 8 | 23 | 32 |
| **TF32** | 1 | 8 | 10 | 19 |
| FP16 | 1 | 5 | 10 | 16 |

### Key Properties

- **Same exponent bits as FP32** → Same dynamic range
- **10 mantissa bits** → Same precision as FP16
- **19 bits total** → Tensor Core can process 2 TF32 ops per clock

## Why TF32 Exists

### The Problem

FP32 is accurate but slow.
FP16 is fast but has limited range (requires loss scaling).

### The Solution

TF32 = FP32 range + FP16 precision + FP16 speed

- No loss scaling needed
- Drop-in replacement (lazy format)
- Just replace FP32 with TF32 in most cases

## How It Works (Lazy Format)

TF32 is a "lazy" format — the GPU stores everything as FP32 internally, but Tensor Cores perform math in TF32.

```
Memory: FP32 (32-bit)
Compute: TF32 (19-bit)
```

This means:
- No precision loss when storing weights/activations
- Speedup comes from Tensor Core matrix multiplication
- Easy migration: just change data type

## Usage in Practice

### Easiest: Just Use It

In PyTorch with Ampere+ GPUs:
```python
# Default: TF32 enabled for Tensor Cores
output = model(input)  # Automatically uses TF32 for matmul
```

### Explicit Control

```python
import torch

# Check if TF32 is enabled
print(torch.backends.cuda.matmul.allow_tf32)  # Default: True

# Disable TF32 for testing
torch.backends.cuda.matmul.allow_tf32 = False
torch.backends.cudnn.allow_tf32 = False
```

### When TF32 Matters

TF32 primarily affects:
- Matrix multiplications (Tensor Core operations)
- Convolutions (via cuDNN)
- Linear layers (nn.Linear)

It does NOT affect:
- Element-wise operations
- Softmax, layernorm
- Memory operations

## Comparison with Other Formats

| Format | Range | Precision | Speed | Loss Scaling | Use Case |
|--------|-------|-----------|-------|--------------|----------|
| FP32 | Full | Highest | Slow | ✗ | Gradient computation |
| **TF32** | Full | High | Fast | ✗ | Training (easiest) |
| BF16 | Full | Medium | Fast | ✗ | Training (stable) |
| FP16 | Limited | High | Fastest | ✓ | Inference |

## TF32 vs BF16

Both have FP32 range. Key differences:

| Aspect | TF32 | BF16 |
|--------|------|------|
| Precision | 10 bits | 7 bits |
| Storage | 32-bit | 16-bit |
| Memory usage | Higher | Lower |
| Use in activations | FP32 storage | Can be BF16 |
| Popularity | NVIDIA only | Cross-vendor |

**Recommendation**:
- Use TF32 if you want zero-effort speedup
- Use BF16 if you want smaller memory footprint
- Use both if you need maximum efficiency

## Performance Impact

Typical speedup from FP32 → TF32:
- Matrix multiplication: **~3-8x** (depends on size)
- End-to-end training: **~1.5-2x**

The speedup is less than theoretical because:
- Non-Tensor Core operations still use FP32
- Memory bandwidth may be bottleneck
- Small layers can't saturate Tensor Cores
