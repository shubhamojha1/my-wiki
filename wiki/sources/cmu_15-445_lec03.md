## Summary

Lecture 3 covers how databases store data on disk. It introduces the disk-oriented DBMS architecture, explaining how the storage manager organizes data into pages and files, and how the DBMS manages data movement between disk and memory.

## Disk-Oriented DBMS Architecture

Most DBMSs use a **disk-oriented** architecture where the primary storage location is non-volatile disk, not DRAM. This is in contrast to **memory-oriented** DBMSes where tables live entirely in memory.

### Design Goals

- Handle databases larger than available memory
- Minimize disk I/O stalls
- Maximize sequential access over random access

## File Storage

The DBMS stores the database as one or more files on disk:
- Some use a **file hierarchy** (multiple files for different tables)
- Others use a **single file** (e.g., [[SQLite]])

## Database Pages

A **page** is a fixed-size block of data that contains database content:
- Tuples (row data)
- Indexes
- Metadata

Most systems don't mix different types within pages. Some require pages to be **self-contained** (all info needed to read the page is on the page itself).

Page sizes are typically 4KB-16KB (e.g., PostgreSQL 8KB, MySQL 16KB, Oracle blocks).

## Database Heap

A **heap file** is an unordered collection of pages containing tuples:
- No inherent ordering
- Need to scan entire file to find a tuple (unless indexed)

## Storage Manager

The **storage manager** is the DBMS component responsible for:
- Maintaining a database's files
- Managing data movement between disk and memory (the **buffer pool**)

## DBMS vs OS

Unlike the OS, a DBMS provides:
- Different page sizes than OS blocks
- Access methods beyond file read/write
- Non-volatile writes with WAL (Write-Ahead Logging)
- Bug-free I/O code
