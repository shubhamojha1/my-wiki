## Summary

Lecture 4 dives deep into buffer pool management—the core of how a DBMS manages memory. It covers buffer pool organization, metadata structures, replacement policies, and the trade-offs between DBMS-managed buffers and OS memory-mapped I/O.

## Buffer Pool Organization

- **Frames**: Fixed-size slots in memory (array entries), each holding one database page
- **Pages**: When requested, an exact copy of the disk page is placed into a frame
- **Write-back cache**: Dirty pages are buffered and not written to disk immediately

## Buffer Pool Metadata

- **Page Table**: Hash table tracking pages currently in memory
  - Maps page_id to frame location
  - Tracks pin count (how many threads are using the page)
  - Tracks dirty flag (modified but not written)
  - Protected with latches for thread safety

## Key Operations

- **Pin**: Increment pin count, prevent eviction
- **Unpin**: Decrement pin count, allow eviction
- **Flush**: Write dirty pages to disk

## Buffer Replacement Policies

- **LRU**: Evict least recently used page
- **Clock**: Approximate LRU using reference bit (circular buffer)
- Most DBMSs use LRU or CLOCK

## Memory-Mapped I/O

- OS manages page faults, loads data into physical memory
- **Problem**: OS may evict DBMS pages unexpectedly
- **Trade-off**: Simpler implementation but less control

## Other Memory Pools

DBMS also needs memory for:
- Sort buffers
- Join hash tables
- Query caches
- Metadata
