## Summary

Lecture 10 covers how to safely allow multiple threads to access indexes concurrently. It introduces latches (vs locks), latch implementations, and B+Tree concurrency protocols.

## Locks vs Latches

### Locks
- **Database-level**: Protect logical data
- **Long duration**: Held until transaction ends
- **May rollback**: Can be rolled back
- **Deadlock detection**: Needed

### Latches
- **Data structure-level**: Protect internal structure
- **Short duration**: Held only during operation
- **No rollback**: Cannot be undone
- **Deadlock prevention**: Design prevents deadlock

## Latch Implementations

### Test-and-Set (TAS) Latch
- Atomic instruction
- Fast but high contention
- Good for few threads

### Mutex
- OS-provided
- Simple but slow

### Reader-Writer Latch
- Multiple readers allowed
- Writers get exclusive access
- Queue fairness options

## B+Tree Concurrency

### Latch Crabbing/Coupling Protocol

**Read path**:
1. Get parent latch
2. Get child latch
3. Release parent latch (if safe)

**Write path**:
1. Get parent latch
2. Get child latch
3. If child full, split/merge
4. Propagate up if needed
5. Release all latches

### Optimizations

- **Read-only scans**: Don't need latches if no modification
- **Leaf node scans**: Link pointers, no parent needed
- **Latch coupling order**: Always parent before child
