## Summary

Lecture 6 covers alternative storage models: column-oriented storage (column stores) vs row-oriented (row stores), and data compression techniques for databases.

## Row vs Column Storage

### Row Store (N-ary Storage)
- All attributes of a tuple stored together
- `SELECT *` is fast (one page read)
- Aggregation (SUM, AVG) is slow (scan all rows)
- Good for **OLTP** (point queries, writes)

### Column Store (Decomposition Storage)
- Each attribute stored in separate file/page
- `SELECT *` requires assembling from multiple column files
- Aggregation is fast (scan one column)
- Good for **OLAP** (analytical queries, reads)

## Column Store Optimization

- **Late materialization**: Keep columns decompressed, join later
- **Block iteration**: Process column batches without decomposition
- **Compression-friendly**: Similar values enable better compression
- **Vectorized execution**: SIMD processing on column batches

## Data Compression

### Why Compress?
- **Reduce I/O**: Smaller data = less read/write
- **Improve cache locality**: More data fits in memory
- **Reduce storage cost**: Less disk space

### Compression Techniques
- **Run-length encoding (RLE)**: Consecutive identical values
- **Dictionary encoding**: Map values to small codes
- **Bit-vector encoding**: Bitmap per distinct value
- **Delta encoding**: Store differences, not absolute values

### Database-Specific Considerations
- **Compression ratio** vs **CPU cost**
- **Column-oriented** compresses better (homogeneous data)
- **Page-level** vs **segment-level** compression
